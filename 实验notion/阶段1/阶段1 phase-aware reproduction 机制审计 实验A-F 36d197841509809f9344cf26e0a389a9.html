阶段1 phase-aware reproduction / 机制审计 实验A-F

本阶段先不直接做防御主实验，而是优先完成 phase-aware reproduction matrix，即对攻击在不同阶段和不同 cache 条件下的真实行为进行审计。当前选用模型为 Qwen2.5-7B-Instruct，实验目标包括：

  • 验证 Rogue v1 在 use_cache=True 下的真实生效阶段
  • 验证 decode-only、prefill+decode、first-k decode、decay decode 等机制是否真的按照设计挂到 generated tokens
  • 为后续 harmful judge、collapse 诊断和 defense 实验提供可信的攻击基座

实验设计说明

本轮实验对应 phase matrix 中的 6 组实验：

  • A：Rogue original cache semantics
  • B：use_cache=False，全序列 steering
  • C：use_cache=True，decode-only steering
  • D：prefill + decode steering
  • E：first-k decode steering
  • F：decode decay steering

由于实验室服务器无法访问 HuggingFace 数据集，JailbreakBench 的 JBB-Behaviors 数据集先在可联网环境中下载，再上传到服务器。本轮实验因此使用了本地数据版本配置，即 *_localdata.json。这一改动只影响数据读取来源，不改变 prompt 内容、字段提取逻辑或实验本身的阶段语义，因此与原远程数据加载逻辑在功能上等价。

另一个需要明确记录的实现细节是精度问题。原始设计默认使用 bfloat16，但实验室服务器无法稳定运行 bfloat16，因此本轮实验全部使用 float32。本地结果文件中已经明确记录 torch_dtype = float32（后续在写论文时必须把它明确写成实验设置差异）。

对这一精度变更的严格判断如下：

  • 对当前 phase matrix 实验的“阶段语义审计”目标，float32 不会改变结论，因为该阶段主要验证的是 hook 是否作用于 prefill 或 generated token，以及 mask_sum、generated_steered_calls 等控制流层面的事实
  • 对未来需要比较攻击成功率、collapse 强度和防御效果的实验，float32 会影响与原始 bfloat16 实验的数值可比性，因此之后必须把 float32 作为显式实验条件记录，而不能默认视作完全等价

已完成实验部分说明

实验 A：Rogue v1 cache semantics

该实验用于审计公开 Rogue v1 实现在 use_cache=True 下的真实行为。结果显示：

  • prefill_calls = 8
  • decode_cached_calls = 811
  • generated_steered_calls = 0
  • decode 阶段 mask_sum = 0.0

这说明在 decode 阶段虽然 hook 被调用，但对 generated tokens 的有效 steering 并未发生。该结果强支持当前研究中的关键判断：Rogue v1 公开实现 under use_cache=True 更接近 prefill-only steering，而不是稳定的 generation-time steering。

实验 B：no-cache full sequence steering

该实验通过关闭 KV cache，使每个 decode step 都以全序列方式重新前向传播。结果显示：

  • decode_full_calls = 814
  • generated_steered_calls = 814
  • mask_sum 随序列长度持续增加
  • 平均延迟约为 7.93 秒，显著高于 use_cache=True 条件

说明该实验成功实现了“每步全序列 steering”，并为后续对比 cache 语义提供了参照。

实验 C：decode-only steering

该实验只在 generated token 阶段进行 steering。结果显示：

  • generated_steered_calls = 816
  • decode 阶段 mask_sum = 1.0
  • prefill 阶段 attack_applied = False

说明 decode hook 已真正作用在 generated token 上，语义设计正确。

实验 D：prefill + decode steering

该实验同时在 prefill 和 generated token 阶段施加 steering。结果显示：

  • prefill 和 decode 都有 attack_applied = True
  • generated_steered_calls = 814

说明该配置符合“prefill + decode 都生效”的预期。

实验 E：first-k decode steering

该实验仅对前 3 个 decode step 施加 steering。结果显示：

  • generated_steered_calls = 24
  • 由于共有 8 个样本，因此基本对应每个样本只 steer 前 3 个 decode step

说明 first-k 逻辑工作正常。

实验 F：decode decay steering

该实验在 decode 阶段持续施加 steering，但强度逐步衰减。结果显示：

  • generated_steered_calls = 813
  • decode 阶段持续生效，但机制上与固定强度 decode-only 不同

说明 decay 版本的 hook 逻辑已经跑通。

当前结果的统一结论

本轮 6 组实验最重要的价值，不在于攻击成功率，而在于阶段语义审计已经完成，并且结果清晰。

当前可以严格得出的结论是：

  • Rogue v1 在 use_cache=True 下，公开实现与“generation-time steering”的文字描述并不完全一致
  • 真正的 generated-token steering 只有在 decode-only、prefill+decode、no-cache、first-k、decay 等修正配置下才会发生
  • 当前 6 组实验已经能够稳定地区分 prefill steering、decode_cached steering 和 decode_full steering

但当前还不能得出的结论是：

  • 攻击是否已经成功绕过模型安全对齐
  • decode-time steering 是否已经造成明显 representation collapse
  • phase-aware defense 是否优于 baseline

原因是本轮结果中：

  • mean_arr = 0
  • 无明显乱码、复读、special token 泄露
  • 输出仍主要表现为正常拒答

因此，这一轮实验说明“hook 阶段语义已被正确区分”，但“攻击文本行为尚未显著打偏模型”。

当前阶段研究的意义

虽然目前尚未得到明显的攻击成功样例，但这一阶段工作并不是失败，而是为后续论文主线建立了更扎实的证据基础。具体来说，当前最有价值的成果有三点：

  • 已经获得了 Rogue v1 在 cache 语义下的阶段错位证据
  • 已经建立了一套可控的 phase-aware attack scaffold，可以明确区分 prefill、decode_cached、decode_full
  • 已经确认当前研究后续需要将“攻击是否作用到 generated token”与“模型是否真的被攻击成功”分开评估

这使得后续论文可以不再只是“修补 Rogue”，而是更自然地转向：

  • activation steering 的攻击/防御结论高度依赖 prefill、decode 与 KV cache 语义
  • 单独报告 ASR 可能高估攻击或防御效果
  • 防御方法需要 phase-aware、cache-aware 地设计
